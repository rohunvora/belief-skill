belief.board architecture

The prediction verification layer for the internet

The Problem

Predictions are everywhere. Verification is nowhere.

Everyone makes predictions — finance Twitter, sports bettors, political pundits, podcast hosts. But there's no canonical place to verify who was right. Screenshots are fakeable. Memory is selective. Track records don't exist.

The Insight

SKILL.md taught agents new capabilities.
TRADE.md teaches the board new domains.

Just like anyone can write a SKILL.md to extend what Claude can do, anyone can write a TRADE.md to extend what kinds of predictions the board can handle. The spec file is the extension mechanism — not code changes.

SKILL.md

  • Frontmatter: name, triggers, description
  • Body: instructions, constraints, output format
  • Scripts: deterministic operations
  • References: detailed docs (loaded on demand)
  • Effect: Claude learns a new capability
→

TRADE.md

  • Frontmatter: domain, resolution type, sources
  • Body: position schema, resolution rules, display
  • Scripts: validate, resolve, fetch prices
  • References: domain knowledge (instruments, leagues)
  • Effect: Board handles a new prediction type

The principle: The spec file IS the extension point. Adding crypto DeFi predictions, weather predictions, or startup outcome predictions doesn't touch the board's core code. A new domain author writes one TRADE.md and optionally some scripts. The board reads it and knows what to do.

File Structure

Each domain is a self-contained plugin

The board discovers domains by scanning the domains/ directory. Each has a TRADE.md manifest and optional scripts for validation, resolution, and live pricing.

board/ ├── server.ts ← universal envelope handler ├── db.ts ← one table, domain-agnostic ├── types.ts ← Prediction envelope (9 fields) └── domains/ ├── finance/ │ ├── TRADE.md ← schema + rules + display │ └── scripts/ │ ├── validate.ts ← is this a valid finance prediction? │ ├── resolve.ts ← check outcome against price │ └── price.ts ← live price for active predictions ├── sports/ │ ├── TRADE.md │ └── scripts/ │ ├── validate.ts │ └── resolve.ts ← fetch box score, compare to line └── politics/ ├── TRADE.md └── scripts/ └── resolve.ts
How the board uses TRADE.md

POST /api/predictions

Caller sends { claim, domain, meta: {...} }

Route by domain

Board reads domains/{domain}/TRADE.md frontmatter

Validate meta blob

Run domains/{domain}/scripts/validate.ts against the meta

Insert into universal table

9 envelope columns + JSON meta blob. One table for all domains.

Resolution tick (periodic)

For each domain: run scripts/resolve.ts on active predictions. Update status to won/lost/expired.

Render by domain

Feed, card, and detail page use domain's display rules from TRADE.md

Finance TRADE.md

domains/finance/TRADE.md
---
domain: finance
resolution_type: price
resolution_cadence: realtime
sources:
  - yahoo-finance
  - hyperliquid-api
  - kalshi-api
---

# Position Schema
ticker    string  required
direction long|short  required
instrument string  required
platform  string  required
entry_price number required

# Resolution
Won = price moved 20%+ in direction
Lost = hit kill condition or 20%+ against
Expired = past resolve_by, no move

Sports TRADE.md

domains/sports/TRADE.md
---
domain: sports
resolution_type: threshold
resolution_cadence: event
sources:
  - espn-api
  - nba-stats
---

# Position Schema
player    string  required
league    string  required
stat      string  required
line      number  required
direction over|under  required
game_date ISO date  required

# Resolution
Won = actual stat exceeds line (over)
Lost = actual stat under line
Voided = player DNP / postponed
The Typed Envelope

9 universal columns. Everything else in the blob.

The envelope answers every question the UI needs to ask. The blob holds domain-specific detail that only matters on the detail page.

Column Type Purpose Used by
id TEXT PK Unique identifier all
claim TEXT NOT NULL The prediction in plain language feed card
author_id TEXT NOT NULL Who made it (FK → users) feed leaderboard
domain TEXT NOT NULL "finance" / "sports" / "politics" feed routing
status TEXT NOT NULL "open" / "won" / "lost" / "expired" / "voided" feed leaderboard
confidence REAL 0.0–1.0 (nullable) leaderboard
source_url TEXT Where the claim came from card
created_at TEXT NOT NULL When the prediction was made feed
resolved_at TEXT When it was settled leaderboard
meta TEXT (JSON) Domain-specific blob, validated by TRADE.md detail

Design rule: Columns answer list questions (feed, filter, leaderboard). Blob answers detail questions (one prediction at a time). If you're only ever reading it for one specific prediction, it goes in the blob.

Universal metric

Accuracy is universal. PnL is domain-specific.

Won/lost works for everything: stocks, sports props, elections. "73% accuracy over 400 predictions" is meaningful across all domains. PnL is computed by each domain's resolve.ts and stored in the blob — never as a top-level column.

Buyer Tiers

Who pays for the data

Tier 1: Crypto funds & trading desks

Real-time feed of "what is CT translating beliefs into right now" — not sentiment, specific instruments with reasoning. Wintermute, GSR, Amber.

Tier 2: Quant alternative data buyers

Historical database for backtesting, then live feed if signal proves. Point72, Citadel, Two Sigma. $500K–$5M/year per unique data source.

Tier 3: Prediction market platforms

"What markets should we create?" Routing convergence = demand signal. Kalshi, Polymarket.

Tier 4: Broker-dealers and platforms

"Trending beliefs" feature — more predictive than trending tickers. Robinhood, eToro, Coinbase.

The revenue stack: (1) Strategy builders monetize their picks — you take 10%. (2) Data buyers pay for the database — unique alternative data. (3) Prediction platforms pay for demand signals. Three layers, each unlocked at a different gate milestone.
